NG-SOAR Project Documentation

NG-SOAR is a unified security orchestration console built by extending SOARCA-GUI, integrating CACAO playbook discovery, Roaster authoring, SOARCA execution, manual input handling, and durable execution-summary reporting.

Project Overview

The project goal is to provide one professional browser interface for SOAR operations: finding CACAO playbooks, inspecting metadata, editing in Roaster, executing through SOARCA, monitoring runs, handling manual input, and keeping durable last-execution summaries available for dashboards and playbook pages.

Primary UI

SOARCA-GUI as NG-SOAR

The main frontend is apps/SOARCA-GUI. NG-SOAR features are layered into the existing React application instead of using a separate frontend shell.

Authority

SOARCA owns execution

SOARCA remains the authoritative service for CACAO playbook CRUD, validation, execution triggers, reporter state, and manual runtime input.

Authoring

Roaster is the editor

CACAO Roaster is exposed as the user-facing Playbook Editor through /roaster and /roaster/playbook/:playbookId.

Extension

Small NG-SOAR API

The Node/Express BFF persists execution summaries, identities, platform health, and OpenAPI/Swagger documentation without becoming a second SOAR backend.

Architecture Decisions

Decision 1: one frontend The former standalone ng-soar-frontend has been removed. The product surface is SOARCA-GUI extended with NG-SOAR routes, branding, dashboard, playbook search, Roaster hosting, settings, and execution-summary integration.
Decision 2: SOARCA is source of truth Playbooks should be created, read, updated, deleted, validated, and executed through SOARCA APIs. NG-SOAR must not create a duplicate persistent playbook store.
Decision 3: Roaster remains specialized NG-SOAR should not rebuild Roaster's CACAO visual authoring capabilities. It should provide navigation, context, execution status, and a clean handoff into Roaster.
Decision 4: summary persistence is narrow The NG-SOAR API stores durable summary-level execution records because SOARCA reporter data may be temporary. It does not own execution semantics.

Runtime Architecture

The Docker stack presents a single public entrypoint through Nginx. Internally, requests are routed to the frontend, SOARCA, the NG-SOAR API, or Roaster. MongoDB and Mosquitto support persistence and SOARCA runtime messaging.

Frontend Architecture

The frontend is a React + Vite + TypeScript application using React Router, React Query, styled-components, and shared SOARCA-GUI components. NG-SOAR additions are grouped under apps/SOARCA-GUI/src/ng-soar where practical.

Implemented NG-SOAR modules

  • api/executionSummaries.ts for persisted execution state.
  • api/identities.ts for CACAO identity display resolution.
  • api/platform.ts for platform health and OpenAPI reads.
  • dashboard/ for inventory, execution, and health metrics.
  • playbooks/ for search, metadata badges, table view, and version metadata.
  • roaster/ for iframe hosting and execution side panel.
  • settings/ for platform operations and API explorer panels.

Frontend responsibilities

  • Present a coherent NG-SOAR dashboard and navigation model.
  • Load playbooks from SOARCA and enrich metadata client-side.
  • Search across extracted metadata and full CACAO JSON.
  • Display last execution status from NG-SOAR API summaries.
  • Keep SOARCA-GUI monitoring and manual input workflows intact.
  • Host Roaster as the Playbook Editor without duplicating its editor logic.

Backend and API Architecture

Backend responsibility is intentionally split. SOARCA is the SOAR authority. The NG-SOAR API is a compact extension service for summary persistence, identities, platform health, and documentation.

SOARCA

Playbooks and execution

  • GET /api/soarca/playbook/ lists playbooks.
  • GET /api/soarca/playbook/:id reads a playbook.
  • POST /api/soarca/playbook/ creates/imports playbooks.
  • PUT /api/soarca/playbook/:id updates playbooks.
  • POST /api/soarca/trigger/playbook/:id starts execution.
  • /api/soarca/reporter exposes execution reports.
  • /api/soarca/manual/continue handles manual runtime input.
NG-SOAR API

Extension persistence

  • GET /api/ng-soar/health checks API and MongoDB connectivity.
  • GET /api/ng-soar/platform/health aggregates service health.
  • GET/POST /api/ng-soar/execution-summaries reads or persists summaries.
  • GET /api/ng-soar/playbooks/:id/last-execution returns last summary.
  • GET/POST/PUT/DELETE /api/ng-soar/identities manages CACAO identities.
  • GET /api/ng-soar/openapi.json exposes API documentation.
Boundary rule SOARCA owns CACAO playbooks and execution lifecycle. NG-SOAR API owns only NG-SOAR extension metadata: durable execution summaries, identity display records, platform health aggregation, and API documentation.

Key Data Flows

These flows show how the frontend composes SOARCA-owned data with NG-SOAR extension records while keeping service ownership clear.

Dashboard loading

1 User opens dashboard

Browser loads /dashboard from SOARCA-GUI.

2 Load playbooks

Frontend queries SOARCA playbook APIs.

3 Load execution state

Frontend reads SOARCA reporter state and NG-SOAR execution summaries.

4 Load health

Frontend reads SOARCA status and NG-SOAR platform health.

5 Render metrics

Dashboard presents inventory, execution, attention, and platform metrics.

Playbook editing and execution

1 Find playbook

User searches and filters the SOARCA-backed playbook library.

2 Open detail

Detail page combines CACAO metadata, identities, and last execution status.

3 Open editor

User navigates to /roaster/playbook/:playbookId.

4 Edit or validate

Roaster provides the CACAO authoring and validation surface.

5 Execute and monitor

SOARCA executes; monitoring views handle progress and manual input.

Execution summary persistence

1 SOARCA reports state

SOARCA exposes runtime reports through its reporter API.

2 BFF syncs

NG-SOAR API polls reporter state on a configured interval.

3 Normalize

Reporter data is converted into durable execution summary records.

4 Persist

Summaries are upserted into the NG-SOAR MongoDB database.

5 Display

Dashboard, playbook list, detail, and editor panel show last status.

Current Limitations and Roadmap

Known limitations

  • Roaster is embedded/proxied rather than fully merged as native React components.
  • Versioning currently relies on CACAO ID, modified datetime, and optional labels.
  • Execution-summary persistence is summary-level, not a full audit/event store.
  • Authentication and authorization are planned but not complete production hardening.
  • Some dashboard and workflow preview polish remains future UI work.

Future architecture considerations

  • Add API-key protection for mutating NG-SOAR API endpoints.
  • Move toward OIDC and role-aware authorization for production-like use.
  • Expand version history if SOARCA cannot represent all version needs natively.
  • Extract reusable Roaster exports such as PNG and STIX 2.1 wrapping.
  • Support external sharing targets such as TAXII, OpenCTI, and MISP.
